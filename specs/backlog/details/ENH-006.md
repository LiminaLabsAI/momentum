# ENH-006: Add Group Execution Pattern to `brainstorm-project.md`

## Problem

`core/commands/brainstorm-project.md` step 7 says:

> "Create Phase 0 files using the Group Execution Pattern"

But it doesn't define what the Group Execution Pattern is. The command references it without explaining it, so when a user runs `/brainstorm-project` on a fresh project, the agent has no instruction on how to structure `plan.md`.

`core/commands/brainstorm-phase.md` correctly includes the full Group Execution Pattern section. `brainstorm-project.md` should match.

## What's Missing

The following section needs to be added to `brainstorm-project.md`, matching the version in `brainstorm-phase.md`:

```markdown
## Group Execution Pattern for plan.md

Declare the execution order at the top of every plan.md:

\`\`\`
# Sequential:  Group 0 → Group 1 → Group 2
# Parallel:    (Groups 0 + 1 + 2 in parallel) → Group 3
# Mixed:       Group 0 → (Groups 1 + 2 in parallel) → Group 3
\`\`\`

Every group header declares:
- `**Sequential.**` or `**Parallel with Groups X and Y.**`
- External dependencies (libraries, services, running processes)
- Commit message for the group

Standard layout:
- **Group 0** — contracts, types, migrations (sequential, blocks everything)
- **Middle groups** — independent feature areas (parallel candidates)
- **Second-to-last** — wiring and integration (sequential)
- **Last** — verification: tests, benchmarks, smoke tests (sequential)
```

## Fix

In `core/commands/brainstorm-project.md`:

1. After step 7 ("Create Phase 0 files"), add a reference: `(see Group Execution Pattern below)`
2. Append the full Group Execution Pattern section at the bottom of the file — identical to the section in `brainstorm-phase.md`

This ensures both entry-point commands produce consistent, well-structured `plan.md` files.
