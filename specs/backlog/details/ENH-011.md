# ENH-011: Upstream Rules 10 + 11 to CLAUDE.md Template

## Context

cerebrio-sapience developed two additional rules beyond momentum's 9-rule template that are generic and benefit any project:

### Rule 10: Architecture Specs Stability — Additive vs Decisional Changes (monorepo)

Files under `specs/architecture/` are constitutional documents. The key distinction is between **additive bookkeeping** and **architectural decisions**:

**During phase implementation (both types — no spec changes):**
- READ specs as stable reference
- NEVER modify them based on implementation discoveries
- Log all gaps and changes as `[ARCH_CHANGE]` in phase history with `Affects-specs:`

**At phase completion (via `/sync-docs`):**
- Additive changes (new fields, new ports, new modes, new entries extending an existing design): update specs directly — no ADR required
- Architectural decisions (approach changes, trade-off choices, design direction changes): require ADR amendment before spec update

**Why the distinction matters:**
The original rule ("only update via ADR amendment") worked during early phases when the architecture was stabilizing and every change was a decision. By mid-to-late phases, the architecture is proven — most changes are additive extensions, not decisions. Requiring ADRs for bookkeeping creates spec staleness while adding no value. ADRs are for capturing *why* a path was chosen; they're not required when you're just recording *what was added*.

**Summary of the rule:**
- No spec changes during implementation → stable reference while coding
- At phase completion: additive changes go directly into specs; decisional changes go through ADR first
- The history log's `[ARCH_CHANGE]` entries are the source of truth for what changed within a phase

This applies to any monorepo with architecture specs — not cerebrio-specific.

### Rule 11: Evaluator Discipline — Lock Evaluators Before Loops

Before building any learning or optimization loop:
- Define the evaluation set (fixed corpus with known-good outputs)
- Define the scalar (single number that improves or doesn't)
- Commit the evaluator to `tests/benchmarks/` with a version tag
- Build the loop AFTER the evaluator is committed
- NEVER change the evaluator while the loop is being optimized

This applies to any project with ML/AI optimization loops — not cerebrio-specific.
