# ENH-061: Project Preferences

## Problem

Today's recipe templates bake project-specific assumptions into the **global** recipe text, so every downstream install inherits assumptions that hold for momentum (an npm package on GitHub) but are wrong for almost any other project.

Concrete leaks observed:

| Recipe leak | Wrong for |
|---|---|
| `/start-phase` hard-stop text: `… and run npm publish --access public` | Next.js/Astro web apps (deploy, don't publish); Python (pypi); Rust (crates.io); Go (`go build`); .NET (nuget) |
| `/complete-phase` step 9: `gh release create vX.Y.Z …` | GitLab (`release-cli`), Bitbucket (no native releases), Gitea/Forgejo |
| `/validate` + `/start-phase` verification commands: `npm test`, `npm run build` | Python (`pytest`/`uv run`/`poetry`/`pip`), Rust (`cargo test`), Go (`go test ./...`), .NET (`dotnet test`) |
| `/complete-phase` implicit assume: git remote is GitHub | Self-hosted GitLab, Bitbucket Cloud, Forgejo/Gitea, bare ssh remotes |

BUG-023 (filed same day) is the immediate doc-fix for the `/start-phase` npm-publish leak — drop the project-specific clause from the global template so the gate text is project-agnostic ("merge + tag + GitHub Release" remains, since `gh` is the documented default; non-GitHub projects override under `## Project Extensions`).

ENH-061 is the **structural fix**: build a structured preferences surface that recipe templates *read* at execution, so the gate copy + commands ADAPT to the actual project instead of asking the agent to recognize mismatches and improvise.

## Scope (cross-cutting)

| Surface | Role | Notes |
|---|---|---|
| `/start-project` | AUTHORS preferences during the founding step | Natural extension of ENH-060's Phase-25 foundation step — same authoring point, same gate. Wilts if attempted before Phase-25 contract is shipped (Phase 25 already landed v0.32.0, so this is unblocked). |
| `momentum init` | WRITES inferred defaults | Detect project language/forge from manifest files (`package.json`, `pyproject.toml`, `Cargo.toml`, `go.mod`, `*.csproj`) + `git remote get-url origin`; write a default `specs/preferences.md` for the user to confirm/edit at `/start-project` time. |
| `/brainstorm-phase` | READS preferences (verification defaults) | Phase-verification section picks the right test/build commands per `language`/`framework` instead of `npm test`. |
| `/start-phase` | READS preferences (release gate copy) | Hard-stop text uses `publish_target`/`release_flow` to phrase the gate ("merge + tag + GitHub Release + `npm publish`" for npm+npm-publish; "merge + tag + GitHub Release + Vercel deploy" for nextjs+deploy-only; etc.). |
| `/complete-phase` | READS preferences (release commands) | Step 9 picks `gh release create` (GitHub) vs `glab release create` (GitLab) vs platform-agnostic fallback; step 10 runs project-specific publish/deploy ONLY when configured + approved. |
| `/validate` | CHECKS preferences invariant | Founded ⟹ preferences file exists & matches lifecycle state; flagged if drifted from `package.json`/etc. (`momentum upgrade` re-infers if needed) |
| `/track` / `/review` | Unaffected | Open scope question: should a new project with preferences drive the default work-type hint? (leave out v1; not load-bearing for the release-gate fix.) |
| AGENTS.md / CLAUDE.md template | Mentions the preferences file in the navigation table | Quick pointer + example; otherwise unchanged. |

## Options Considered

### Option A — `specs/preferences.md` (markdown, authored)

A short markdown file in the specs tree, same family as `status.md` / `backlog.md`. Authored at `/start-project` time; read at recipe-execution time.

**Pros:**
- Native to momentum's "specs as the source of truth + markdown machine-readable" model (parallels the Phase-25 contract: foundation docs authored, not scaffolded).
- Version-controlled, reviewable, hand-editable — consistent with how every other momentum artifact works.
- No new file format or parser; one tiny reader (`core/preferences.js::readPreferences(specsDir)`).
- Free-form `## Project Extensions`-style bottom section for project-specific knobs that don't fit the enum.
- Pairs cleanly with OKF v0.1 (`type` frontmatter on the file makes it part of the bundle).

**Cons:**
- Markdown is parseable but no schema enforcement — typos in `publish_target: pypi` silently fall back to defaults.
- Slightly heavier to consume than JSON (regex/table parse vs `JSON.parse`).

### Option B — `.momentum/preferences.json` (JSON, machine-only)

Structured JSON file under the lock state directory.

**Pros:**
- Strict schema, trivial to parse, easy to validate with `momentum doctor`.
- Lives next to `installed.json` — already momentum-owned territory.

**Cons:**
- Breaks the "specs/ is the source of truth" principle — `.momentum/` is **state**, not **content**: lock files, lane registries, handoff inboxes. Project preferences are content (decisions made by the human about their project), not state.
- Not part of the OKF bundle; users browsing `specs/` won't see it.
- Encourages a parallel machine-readable layer that diverges from the rest of momentum's single-layer model — sets a precedent for more machine-only files where markdown would do.

### Option C — Extend `installed.json` with a `preferences` field

Stuff preferences into the existing lock file.

**Pros:**
- No new file at all.
- Lazy-detection: `momentum init`/`upgrade` writes the inferred defaults into the lock, recipes read them out.

**Cons:**
- `installed.json` is INSTALLED-STATE (what momentum put on this repo, what agent, what version). It is a manifest of momentum's actions, not a record of the user's project choices. Mixing the two slides the D1 contract (lock file as version-of-record) toward a project-config file.
- Invisible to a user browsing the repo — deeply unintuitive place to look for "where's my publish target set".
- Cannot be authored by the user at `/start-project` time without making `installed.json` user-writable, which defeats its purpose.
- Brews hidden per-version upgrade problems (which lock migrations already complicate).

### Option D — `## Project Preferences` extension block in primary instruction file (CLAUDE.md / AGENTS.md)

Treat preferences as a project-extension block.

**Pros:**
- No new file.
- Same upgrade-safe marker pattern momentum already uses.

**Cons:**
- The primary instruction file is AGENT RULES, not project data. Recipe templates reading a "rules file" for project facts inverts the separation of concerns — and the natural reader is an *agent* deciding whether to ask, not a *recipe* deciding which command to emit.
- Project preferences must be machine-consumable by `bin/momentum.js` itself (so the CLI can pick the right release command). The primary instruction file is markdown-for-agents, not JSON-for-the-CLI; the CLI parsing its own managed CLAUDE.md is awkward.
- Hard for `/validate` to invariant-check.

## Recommendation

**Option A — `specs/preferences.md`**. Same model as `status.md`/`backlog.md`/`history.md`: authored by the human via a recipe, read by recipes + CLI, version-controlled. Matches the Phase-25 contract (foundation docs authored, not scaffolded — preferences are foundation-adjacent). The "no schema" con is mitigated by a tiny `readPreferences()` that fails closed (unknown value → default to "npm-publish on GitHub" with a warning), so typos degrade to current behaviour, not to wrong-actions.

## Sketch — preferences.md shape

```markdown
---
type: Preferences
---

# Project Preferences

> Authored by /start-project. Read by /start-phase, /complete-phase,
> /brainstorm-phase, /validate. See ENH-061.

| Key | Value | Notes |
|-----|-------|-------|
| language | node | inferred from package.json |
| framework | nextjs | inferred next dependency |
| git_forge | github | inferred from git remote |
| publish_target | none | web app — Vercel deploy, not registry publish |
| release_flow | tag-and-deploy | tag + GitHub Release + Vercel deploy hook |
| test_command | npm test | overrides allowed |
| build_command | npm run build | overrides allowed |
| deploy_command | vercel deploy --prod --prebuilt | project-specific |

## Notes

- Vercel deploy runs from CI on tag; momentum does NOT invoke it directly
  (mirrors the npm-publish model: external action, project-extension owned).
```

## Open Questions

1. **Ownership of `/start-project`'s preferences authoring step.** The Phase-25
   contract ships `/start-project` as the *founding* step (authors
   charter/principles/success-criteria/roadmap + status Summary). Adding a
   preferences authoring sub-step there is natural — but does the brainstorm
   ask the operator for preferences explicitly, or infer + confirm? Recommend:
   **infer + present for confirmation** (low friction, easy override).

2. **Forge detection when remote is bare ssh / on-prem GitLab.** `git remote
   get-url origin` alone is ambiguous. Options: ask the operator at
   `/start-project`; document defaults; fallback to `github` (most-common,
   preserves current behaviour). Recommend: **ask at start-project time with
   a sensible default** — not worth detecting exotic forges silently.

3. **Languages without a clear single publish target.** Python has
   `pypi` (per project) and conda-forge/Artifactory for enterprise;
   .NET has nuget.org + private feeds. Should `publish_target` be
   free-form (string) or enum? Recommend: **enum + `custom`** with the
   custom command listed in a free-form `publish_command` field.

4. **Per-agent override.** Should the opencode adapter's recipe overlay be
   able to override a preference (e.g. switch from `gh` to a
   platform-native release command)? Recommend: **no — preferences are
   project-wide, not per-agent.** Per-agent divergence is exactly what
   the multi-adapter contract (Phase 22c, ADR-0007) tries to avoid.

5. **Pair with ENH-060's foundation gate.** The Phase-25 "founded ⟹
   foundation files exist" invariant can be extended to "founded ⟹
   preferences exist" in the same gate. Recommend: **yes — same gate,
   one more file-existence check.** Schedule the preferences landing
   as a follow-on to Phase 25 (Phase 26 Intelligence or Phase 27
   Platform) so the gate is extended in one place rather than
   retrofit.

## Phasing Recommendation

- **Now (P1 — short-term fix):** BUG-023 — drop the `npm publish` clause
  from the global `/start-phase` hard-stop text. Template-only edit,
  fingerprint re-baseline. Restores project-agnostic behaviour
  immediately; preserves the project-extension path the momentum self-repo
  already uses.

- **Phase 26 Intelligence (P1 — structural fix):** ENH-061 — preferences
  surface + recipe-template read sites. Natural pairing with Phase 26's
  "self-healing release gates" deliverable (the gate decides which
  release command to emit based on preferences, not a hardcoded
  assumption). Fits "context-window-aware task sizing" theme only
  loosely; better fit is the "release gates self-heal to the project's
  actual release flow" framing.

- **Alternative: Phase 27 Platform.** If Phase 26 stays narrowly scoped to
  runtime learning / rule evolution, ENH-061 can ride Phase 27 (MCP server
  `/specify`, `/decide`, skill authoring) — the preferences reader is a
  natural first-class recipe-consumed datum, and `/decide` (ADR + decision
  authoring) could be extended to author preferences at start-project.

Operator decision pending: target Phase 26 vs Phase 27 placement.
