# ADR-0010: The Agent Instruction File Is a Projection of `specs/`, Never Authored

## Status

Accepted

## Context

momentum installs a primary instruction file per coding agent — `CLAUDE.md`
for Claude Code, `AGENTS.md` for Codex / Antigravity / opencode. Historically
each file has had two regions:

- a **managed** region (Navigation, Rules 1–15, Naming, Constraints) replaced
  on `momentum upgrade`, and
- a **`## Project Extensions`** region *below a marker* that upgrade **preserves
  verbatim** — the place users hand-author project-specific rules.

This design produces divergence between agents. The self-repo made it concrete:
CLAUDE.md (installed first) and AGENTS.md (opencode added later) drifted apart
for four reasons — two of them bugs:

1. **Staleness.** `momentum upgrade` refreshes only ONE agent at a time
   (`upgrade()` writes `state.agents[agent]`), never iterating all installed
   agents. Upgrading Claude Code left opencode's AGENTS.md behind — it never
   received the Phase-25 "Not founded" paragraph or the Phase-26 config
   navigation row.
2. **Ecosystem pointer gap.** `core/ecosystem/lib/pointer.js` injects the
   `<!-- ecosystem:begin -->` block into only ONE file (it prefers CLAUDE.md),
   so a Codex/opencode session reading AGENTS.md is blind to the ecosystem.
3. **Project Extensions divergence.** Because `## Project Extensions` is per-file
   preserved prose, CLAUDE.md grew a full release checklist + session-start
   self-audits while AGENTS.md stayed a stub. Independent edits → permanent drift.
4. **Intentional per-adapter differences** (`TodoWrite`→"task tool"; the
   AGENTS.md recipe/skills/plugin surfaces) — correct, generated per adapter.

Causes #1 and #2 are mechanism bugs. Cause #3 is *structural*: any per-file
authoring surface is a drift surface by construction. This is the same lesson
Phase 23 (single-sourced the *rules*) and Phase 26 (moved *config* out of the
templates into `specs/config.md`) already taught — the last hand-authored
surface simply hadn't moved yet.

## Decision

**The agent instruction file is a pure projection of `specs/` — it carries no
hand-authored content.** It is 100% generated by momentum: managed rules +
pointers. Every project-specific thing lives exactly once in `specs/`:

- **Structured** facts → `specs/config.md` (read directly by recipes; Phase 26).
- **Prose** rules (session-start audits, constraints, project-specific
  guidance) → a new shared **`specs/project-rules.md`** (OKF type `Project
  Rules`), referenced by an identical managed **pointer** in every instruction
  file's `## Project Extensions` section.
- **Ecosystem** context → the `<!-- ecosystem:begin -->` block, injected into
  **every** present instruction file.

`## Project Extensions` therefore stops being an authoring surface and becomes a
managed pointer. Consequences:

1. **`upgrade` syncs every installed agent** (`installed.json.agents`) — no
   agent's file silently drifts.
2. **`pointer.js` injects the ecosystem block into every instruction file.**
3. **Migrate, never drop.** On upgrade, existing `## Project Extensions` prose is
   MOVED into `specs/project-rules.md` (verbatim, idempotent), then the section
   is replaced by the pointer. A one-time, content-preserving migration — the
   same shape as Phase 24 (OKF) and Phase 25 (placeholder removal). Nothing
   authored is ever lost.

Genuinely agent-specific rules are written in `project-rules.md` with an inline
qualifier (e.g. "*(Claude Code only)*"), exactly as "*monorepo only*" rule
qualifiers already work — no per-file free zone is needed, and re-introducing
one would re-open the drift surface this ADR closes.

## Consequences

- **Positive:** cross-agent consistency becomes structural — there is nothing
  per-file to keep in sync, so nothing can drift. Adding a second adapter to an
  existing project starts consistent by construction. One place to edit project
  rules (`specs/project-rules.md`), version-controlled like the rest of `specs/`.
- **Negative / cost:** a breaking change to the long-standing "`## Project
  Extensions` is your preserved zone" contract — mitigated by the migrate-never-
  drop guarantee. Agents perform one extra file read (`project-rules.md`) — the
  same cost as the Navigation-table pointers momentum already relies on.
- **Migration:** `momentum upgrade` migrates existing installs automatically and
  reports it; downstream projects lose no content.

## Alternatives considered

- **Inject full content into every file (not a pointer).** Rejected: physically
  duplicates content across N files and re-introduces the "sync can miss an
  agent" failure this ADR exists to prevent.
- **Keep a per-file free zone alongside the pointer.** Rejected: a free zone is a
  drift surface by construction; the inline-qualifier pattern covers the rare
  genuinely-agent-specific rule without one.
