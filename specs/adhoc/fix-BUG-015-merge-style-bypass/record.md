## Current Behavior

`COMMIT_SUBJECT_BYPASS` in `core/git-hooks/contract.js` (line 52) exempts only
git's default merge style `^Merge ` (plus `Revert "`, `fixup!`, `squash!`,
`amend!`, `Initial commit`) from Conventional-Commit validation. The CLAUDE.md
Naming Conventions and this repo's entire merge history use lowercase
`merge: x → y`, so the very first live hook-validated merge
(`git merge --no-ff -m "merge: chore/upgrade-self-v0.25.0 → staging"`) stalled
at "Not committing merge". The hook payload ships to every downstream install,
so the same rejection fires on any repo using the house merge style.

## Expected Behavior

The commit-msg hook accepts the house merge style — any subject starting with
`merge: ` (lowercase, colon, space), e.g.
`merge: phase-x → staging (v1.2.3)` — in addition to git's default `Merge `
style. Fix candidate (a) from the backlog: extend the bypass regex with
`^merge: ` (one-line, template-wide, honors existing history). The self-repo's
installed copy at `.githooks/contract.js` is mirrored byte-identical so the
live gate carries the fix.

## Unchanged Behavior

- Conventional-Commit validation for all normal subjects is unchanged — a
  non-conventional subject like `merged stuff` is still rejected (no colon /
  not a bypass prefix; `merge` followed by anything but `: ` does not match).
- Git-generated bypasses are unchanged: `Merge branch 'x'`, `Revert "…"`,
  `fixup!`, `squash!`, `amend!`, `Initial commit` all still pass.
- The accepted Conventional type list (`CONTRACT.conventionalTypes`) is
  untouched — `merge` is NOT added as a type (fix candidate (b) rejected as
  larger than needed).
- All other contract functions (pre-push gates, sentinel, evidence gate) are
  byte-for-byte unchanged.
